// System Architecture

How Every Byte
Flows — In Depth.

Every data path, every API call, every handshake — laid bare. CloudBeat has no hidden servers. Here is the complete wiring diagram of exactly how it works, from the moment you hit play to the moment your friend hears the same song across the world.

🔍
Step 1 — Scan Repo
CloudBeat calls the GitHub API to scan your repository and build the song list.
01
GET api.github.com/repos/{owner}/{repo}/contents
02
Recursively scans all folders
03
Filters: .mp3 .wav .flac .ogg .m4a .aac .opus
04
Builds song objects with raw URLs
📡
Step 2 — Stream Audio
The audio element's src is set to a raw GitHub URL. The browser streams it directly — no proxy, no middleware.
01
audio.crossOrigin = 'anonymous'
02
audio.src = raw.githubusercontent.com/...
03
audio.load() → wait for canplay event
04
audio.play() → stream begins
🎛
Step 3 — EQ Processing
The raw audio stream passes through a 10-band equalizer chain built entirely in the Web Audio API before reaching your speakers.
01
AudioContext.createMediaElementSource(audio)
02
10× BiquadFilterNode (lowshelf → peaking → highshelf)
03
Chain: source → eq[0] → … → eq[9] → destination
04
All processing in-browser, zero latency added
💾
Step 4 — Save State
Your playlists, liked songs, EQ settings, volume, and listen stats are persisted to localStorage — or synced to Google Drive if signed in.
01
Debounced save on every meaningful action
02
localStorage: cb_settings, cb_playlists, cb_stats
03
Drive sync: writes cloudbeat.json to AppData
04
No data ever leaves your own accounts
🏠
Host starts Jam
Host generates a 5-char room code and begins writing their playback state to Firebase on every play, pause, and seek event.
01
Random room code generated (e.g. AB3X7)
02
Code becomes Firebase path: /jam/AB3X7/state
03
Every play/pause/seek → jamWrite() fires
04
Host shares code with listeners verbally / via chat
🔥
Firebase state object
The state written to Firebase is tiny — just enough metadata for listeners to find and seek the same track. No audio bytes go through Firebase.
01
trackName, repoOwner, repoName, repoBranch
02
repoPath (full path of the audio file)
03
position (seconds), paused (bool), ts (timestamp)
04
nickname · email (display only)
👂
Listener syncs
User 2 enters the code, subscribes to that Firebase path, and receives state pushes in real-time via WebSocket. Each push triggers an audio seek with lag correction.
01
User 2 enters code → subscribes via onValue()
02
State arrives → CloudBeat loads host's repo
03
Fetches same audio from GitHub Raw CDN
04
Seeks to: position + (Date.now() - ts) / 1000
🎮
Control handoff
Host can grant playback control to any listener. That listener becomes the writer; the host and all others become readers until control is reclaimed.
01
Host writes controls/{uid}/hasControl: true to Firebase
02
All clients read jamControlledUid on every sync
03
Only the controller's actions trigger jamWrite()
04
Host can revoke at any time
🔐
OAuth 2.0 Sign-in
Google handles the entire authentication flow. CloudBeat never sees your password — only the access token that Google grants.
01
CloudBeat opens accounts.google.com popup
02
You consent to Drive file access
03
Google returns an access token to the browser
04
Token stored in JS memory only — never on disk
📋
File Discovery
CloudBeat calls the Drive API to list your audio and video files. It scans folders you navigate and shows them in the Drive view.
01
GET drive.googleapis.com/v3/files
02
Filter: mimeType contains 'audio/' or 'video/'
03
Recursive folder navigation on demand
04
File metadata only — no content yet
💧
Blob Streaming
When you play a Drive file, CloudBeat fetches it as a binary Blob via the Drive API and creates a local object URL. No audio data passes through any CloudBeat server.
01
fetch Drive file download URL with auth header
02
Response streamed as a Blob
03
URL.createObjectURL(blob) → blob://...
04
audio.src = blob:// → plays locally, no CORS issue
☁️
AppData Sync
Your playlists, liked songs, EQ presets, and settings are saved to a hidden cloudbeat.json file in your Drive's AppData folder — invisible to all other Drive apps.
01
Writes to Drive AppData (appDataFolder scope)
02
Not visible in your regular Drive file list
03
Syncs automatically on sign-in across devices
04
Deletable anytime from Drive settings

// Legal & Ethical Stance

Full
Transparency.

CloudBeat has nothing to hide. Here is everything you need to know about how it operates, what it does, and what it will never do.

Zero Commercial Activity
The owner, designer (Aayan Singh), and anyone affiliated with the creation or maintenance of CloudBeat does not earn, collect, or derive any income, revenue, profit, or monetary compensation — directly or indirectly — from this website or its operation.

This includes advertising, sponsorships, affiliate links, donations in exchange for features, subscription fees, or any other form of financial gain. No monetization exists on this platform. None is planned.
// The creator of CloudBeat shall never make any source of revenue from this website. This is an unconditional, permanent commitment.
Anti-Piracy & Creator Support
CloudBeat strongly and unconditionally opposes piracy in every form. The unauthorized reproduction, distribution, or streaming of copyrighted music is illegal and causes direct harm to the artists, composers, producers, and all creative professionals who make music possible.

This platform is designed strictly for personal-use audio files that you own or have legal rights to play. It is not a substitute for licensed streaming services. If you love a song — please support its creator on Spotify, Apple Music, or by purchasing it.
// Music is art. Art deserves respect. Artists deserve to be compensated for their work. Please support them.
Content Ownership
CloudBeat does not own, host, or distribute any music or media content. It is a browser-based tool that connects to sources you configure — your GitHub repositories, your Google Drive, your Firebase database.

Users are solely and entirely responsible for the content stored in their repositories and drives. CloudBeat is a player, not a publisher. The developer has no access to, visibility of, or control over what users stream.
// CloudBeat never stores, transmits, or profits from user audio content.
Privacy by Design
CloudBeat was built with privacy as a foundational principle, not an afterthought. No user data is sent to any CloudBeat-controlled server — because no such server exists.

Google credentials are processed entirely via Google's own OAuth. Firebase data goes to your database. Drive data goes to your Drive. Listen stats are stored in your browser's localStorage. The developer sees nothing.
// Your data is yours. CloudBeat is a tool, not a data collector.
